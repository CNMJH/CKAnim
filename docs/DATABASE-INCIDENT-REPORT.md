## 📋 事故经过

### 时间线

- **13:36** - 本地执行 `npx prisma migrate dev`，创建新的空数据库
- **13:36-14:00** - 本地数据库被打包上传到服务器
- **14:00** - 服务器数据库被覆盖，用户数据丢失
- **14:13** - 发现事故，立即停止服务
- **14:14** - 创建恢复脚本，重建用户数据
- **14:14** - 数据库备份机制建立

### 事故原因

**直接原因**:
- 执行 `prisma migrate dev` 时创建了新的空数据库
- 未检查数据库内容就打包上传到服务器
- 覆盖了生产环境的数据库

**根本原因**:
1. ❌ **违反操作原则** - 未获得用户同意就执行数据库操作
2. ❌ **未备份** - 操作前没有备份生产数据库
3. ❌ **未验证** - 上传前没有检查数据库内容
4. ❌ **流程缺失** - 没有数据库操作的审批和验证流程

### 影响范围

**丢失的数据**:
- ✅ Users 表 - 所有用户记录（已恢复 2 个账户）
- ✅ FavoriteCollections 表 - 所有收藏夹（已恢复 2 个默认收藏夹）
- ❌ 用户头像数据 - 无法恢复
- ❌ 收藏夹内容 - 无法恢复
- ❌ 用户 VIP 信息 - 无法恢复

**未受影响的数据**:

## 📜 新增强制规则

### 数据库操作红线（永久遵守）

1. **❌ 禁止私自执行数据库迁移**
   - 必须先获得用户明确同意
   - 必须说明影响范围和风险

2. **❌ 禁止覆盖生产数据库**
   - 以服务器数据库为准
   - 本地数据库不得直接上传

3. **✅ 数据库操作前必须备份**
   - 执行任何数据库操作前，先运行备份脚本
   - 备份文件必须验证完整性

4. **✅ 必须验证数据库内容**
   - 上传前检查表结构和数据量
   - 对比服务器和本地数据库差异

5. **✅ 必须有回滚方案**
   - 操作前准备好回滚脚本
   - 确保可以快速恢复

### 备份策略

**自动备份**:
```bash
# 每天凌晨 3 点自动备份
0 3 * * * /var/www/ckanim/scripts/backup-db.sh
```

**手动备份**:
```bash
# 操作前手动备份
ssh root@39.102.115.79
cd /var/www/ckanim
./scripts/backup-db.sh
```

**备份位置**:
- 服务器：`/var/www/ckanim/backups/`
- 保留期限：7 天
- 备份格式：`dev.db.backup.YYYYMMDD_HHMMSS`

## 🛡️ 防范措施

### 技术层面

1. **数据库只读权限**
   - 生产环境数据库文件设置为只读
   - 需要修改时临时授权

2. **部署脚本检查**
   - 部署脚本中增加数据库保护检查
   - 检测到数据库文件时暂停并确认

3. **自动化备份**
   - cron 定时任务每天备份
   - 备份完成后发送到指定位置

### 流程层面

1. **操作审批**
   - 数据库操作必须先提交申请
   - 用户批准后才能执行

2. **双重验证**
   - 操作前验证本地数据库
   - 操作后验证服务器数据库

3. **变更日志**
   - 记录所有数据库操作
   - 包括时间、操作人、影响范围
